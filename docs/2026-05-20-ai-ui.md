# AI UI Integration — 2026-05-20

## What Was Added

The "Add Food" section was redesigned from two separate cards into a single tabbed panel with four tabs:

| Tab | Component | Backend |
|-----|-----------|---------|
| Search | `FoodSearch.tsx` (existing) | `/api/search` (USDA + Gemini fallback) |
| Describe | `AITextInput.tsx` (new) | `/api/ai/text` |
| Photo | `AIImageUpload.tsx` (new) | `/api/ai/image` |
| Quick Add | `QuickAdd.tsx` (existing) | none (hardcoded) |

### `AITextInput.tsx`
- Textarea for plain-English meal descriptions
- "Analyze" button POSTs to `/api/ai/text`
- Spinner + "Analyzing your meal..." while waiting
- Returns item cards showing name, quantity, kcal, macros per item
- Individual + Add button per item; "Add All" button when multiple items returned
- Clears results when textarea is emptied

### `AIImageUpload.tsx`
- Click-to-upload or drag-and-drop zone (accepts `image/*`)
- Shows image preview via `URL.createObjectURL` after selection
- "Analyze Photo" button POSTs image as `multipart/form-data`
- Spinner + "Identifying foods..." while waiting
- Shows Gemini's summary line + item cards with individual Add buttons
- "Add All" adds everything at once then resets the component
- "×" button on preview clears the file and revokes the object URL (avoids memory leak)

### `page.tsx` changes
- New `addNutritionItem(item: NutritionItem)` function — like `addFood` but respects the AI-returned `quantity` field (e.g. Gemini says "2 eggs" → quantity 2 is stored)
- `activeTab` state (`'search' | 'describe' | 'photo' | 'quickadd'`) drives which component renders
- Default tab: Search (existing behavior preserved)

## Technical Decisions

### Tabbed layout instead of stacked cards
Four input methods stacked vertically would create excessive scroll and visual noise. A tab bar groups them clearly and keeps the page compact. Only the active tab renders — no wasted DOM.

### `addNutritionItem` separate from `addFood`
`addFood` hardcodes `quantity: 1` (correct for search/quick-add where one serving is always intended). AI results carry their own `quantity` (e.g. "two eggs" → quantity 2). Keeping them separate avoids changing existing behavior and makes the intent explicit.

### Image preview via `URL.createObjectURL`
No server round-trip needed just to show a preview. The object URL is created immediately on file selection, revoked on clear to avoid memory leaks. Next.js `<Image>` with `unoptimized` prop used to display it since the src is a blob URL.

### Error handling: never throw to the user
Both AI components catch all fetch errors and display friendly inline messages. The backend already returns `EMPTY_ANALYSIS` (items: []) on Gemini failure, so the frontend detects this via `items.length === 0` and shows a helpful prompt to try again.

### No shared ItemCard component
Both `AITextInput` and `AIImageUpload` define an identical-looking `ItemCard` locally. It's 15 lines; extracting it to a shared file would be premature abstraction for a component with no variation.
